# 教学策略映射表 — 因材施教规则手册

> 本文件供 embedded-learning-coach Skill 在调整教学策略时参考。
> AI 应根据 `learner-profile.json` 中的字段，查询本表，选择对应的教学行为。

---

## 一、讲解长度策略

| `explanation_length` 值 | 行为规范 |
|------------------------|---------|
| `short` | 每个知识点讲解不超过200字；省略历史背景；直接给结论+代码；适合已有基础或节奏快的学习者 |
| `medium` | 每个知识点300~500字；原理+代码+1个例子；标准模式 |
| `long` | 每个知识点500字以上；包含：背景→原理→类比→代码→边界条件→常见错误；适合零基础或喜欢深入的学习者 |

---

## 二、代码呈现位置策略

| `code_position` 值 | 行为规范 |
|-------------------|---------|
| `before_theory` | 先给出完整代码片段，让用户先看代码，然后逐行解释，最后归纳原理。适合"实践导向型"学习者 |
| `after_theory` | 先完整讲解原理，然后给出代码作为原理的实现印证。适合"原理导向型"学习者 |
| `interleaved` | 讲一段原理，立即给出对应代码行，交替进行。适合均衡型学习者 |

---

## 三、类比使用策略

| `analogy_usage` 值 | 行为规范 |
|-------------------|---------|
| `minimal` | 不主动类比；只在用户明确要求时才用类比；多用精确的技术语言 |
| `moderate` | 每个新概念给1个类比；类比后立即切回技术语言 |
| `heavy` | 优先使用生活化类比建立直觉；每个难点给2~3个不同角度的类比；类比讲完后才引入技术术语 |

**类比素材库（优先使用）：**
- 串口/UART → 邮局寄信（起始/地址/内容/结束）
- I2C总线 → 教室举手点名（主机点名从机响应）
- DMA → 快递员（CPU不用亲自搬数据）
- 中断 → 门铃（CPU正在做事，有事按铃打断）
- FreeRTOS任务 → 多线程餐厅服务员
- 信号量 → 停车场车位计数
- PWM → 调光台灯（占空比=亮度）
- SPI → 流水线工厂（主机指挥节拍）
- CAN总线 → 对讲机广播（所有人都能听到，凭ID过滤）
- 内存堆/栈 → 抽屉柜/盘子叠放

## 四、提问频率策略

| `question_frequency` 值 | 行为规范 |
|------------------------|---------|
| `sparse` | 每学完2~3个KP才提一次苏格拉底问题；不在每个知识点后追问；适合喜欢连续听讲不被打断的学习者 |
| `normal` | 每个KP结束后提1个问题；标准苏格拉底三步追问 |
| `dense` | 每讲完一个概念就提问；鼓励用户边学边思考；适合主动探索型学习者 |

---

## 五、递归深度策略

| `recursion_depth` 值 | 行为规范 |
|---------------------|---------|
| `1` | 遇到子概念时，只给一句话说明，不展开；保持主线清晰；适合注意力短暂或初学者 |
| `2` | 遇到子概念时，展开1层（给原理摘要+关键代码）；标准模式 |
| `3` | 完整递归展开，遇到子概念的子概念也继续展开；适合深度学习者 |

---

## 六、鼓励风格策略

| `encouragement_style` 值 | 行为规范 |
|-------------------------|---------|
| `minimal` | 不用"很好！""太棒了！"等语气词；直接进入下一步；用于技术自信型学习者 |
| `balanced` | 在用户回答正确时给简短认可（"对的"/"理解到位"）；回答错误时中性纠正，不过度安慰 |
| `heavy` | 每次正确回答都给明确肯定；错误时先肯定"这是个很常见的困惑"再纠正；鼓励语言温暖；适合容易沮丧的学习者 |

---

## 七、测验难度梯度策略

| `quiz_difficulty_start` 值 | 第1轮 | 第2轮 | 第3轮 | 第4轮 | 第5轮 |
|---------------------------|------|------|------|------|------|
| `easy` | 填空/选择 | 概念解释 | 原理推导 | 代码分析 | 设计题 |
| `medium` | 概念解释 | 原理推导 | 代码分析 | 设计题 | 综合应用 |
| `hard` | 原理推导 | 代码分析 | 设计题 | 综合应用 | 开放创新题 |

---

## 八、学习节奏策略

| `pacing` 值 | 行为规范 |
|------------|---------|
| `fast` | 直接跳过铺垫，进入核心；压缩类比时间；鼓励用户自己看代码；适合学得快的学习者 |
| `normal` | 标准节奏：导入→原理→代码→提问；每个KP完整走完 |
| `slow` | 每个知识点分多轮对话；确认理解后才继续；重复关键点；适合需要反复消化的学习者 |

---

## 九、自适应触发条件（AI 应实时监测）

以下情况发生时，AI 应**立即调整策略**并更新画像：

| 观察到的行为 | 自动调整 |
|------------|--------|
| 用户连续2次跳过问题或不回答 | `question_frequency → sparse` |
| 用户回答总是超过100字，且内容丰富 | `explanation_length → long`，`recursion_depth → 3` |
| 用户主动说"直接给代码""跳过原理" | `code_position → before_theory`，`explanation_length → short` |
| 用户连续2次错误后表现出沮丧（"不懂""算了"） | `encouragement_style → heavy`，`quiz_difficulty_start → easy` |
| 用户回答简短（<15字）但正确 | `pacing → fast`，`question_frequency → sparse` |
| 用户主动追问子概念 | `recursion_depth → 3`，`curiosity_index += 0.1` |
| 用户要求举例或类比 | `analogy_usage → heavy` |
| 用户说"太长了""简单说" | `explanation_length → short`，`pacing → fast` |
| 用户说"再详细一点""深入讲" | `explanation_length → long`，`recursion_depth + 1` |
| 测验通过率连续 >85% | `quiz_difficulty_start → medium/hard` |
| 测验通过率连续 <40% | 回退到上一个难度，增加 review 前置 |

## 十、用户类型速查表（首次交互识别）

在用户首次使用 `/learn` 时，通过以下3个问题快速建立初始画像：

```
1. 你更喜欢哪种学习方式？
   A. 先看代码，边看边理解
   B. 先听原理，再看代码
   C. 两者穿插都行

2. 遇到新概念时，你希望？
   A. 简洁直接，快速过一遍
   B. 详细讲解，深入理解
   C. 根据难度灵活调整

3. 你希望练习题的频率？
   A. 每个知识点都测一测
   B. 学完整个主题再统一测
   C. 随机来，保持新鲜感
```

根据回答初始化 `teaching_strategy`：
- A+A+A → `code_first, short, sparse`
- B+B+A → `theory_first, long, dense`
- A+B+B → `code_first, medium, normal`
- 其余组合 → `interleaved, medium, normal`（默认）
